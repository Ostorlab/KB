A non-idempotent webhook handler is one that, after successfully verifying the provider's signature, unconditionally re-executes a state-mutating side effect on every delivery of the same event. Webhook providers (such as Stripe, PayPal, GitHub, or any event-driven SaaS) are contractually allowed to redeliver an event: the provider retries on non-2xx responses, supports manual "resend", and re-attempts across failures. Each redelivery carries the same event identifier and the same payload, and is signed and cryptographically valid. A handler that does not persist and check that event identifier before mutating local state will apply the side effect once per delivery.

The impact is amplified when the mutated state is an authorization-relevant cache that gates entitlement, access, or billing status. A canonical pattern is a payment-success webhook that resets a cached `last_payment` / `paid_until` timestamp to `now()` on every delivery. Because the active entitlement window is computed as `last_payment_cached + <billing period>`, each verified redelivery re-anchors a full fresh billing window forward in time, re-enabling features, scans, or access that the legitimate billing flow had already expired or bounded. When that cached field is the sole trust source and no independent server-side reconciliation re-derives it from a live provider call, the webhook mutation is authoritative and the re-extension is genuine.

A second, independent impact is availability and cost: branches that issue outbound provider API calls (for example `Subscription.list(customer=...)`) on every delivery, without caching or rate-limiting, amplify outbound API usage on every replay or redelivery, exhausting rate-limits and quota.

This defect is distinct from a missing authentication control: signature verification may be effective and correctly ordered before the mutation. The flaw is the absence of idempotency controls (event-ID deduplication, `transaction.atomic()` + `select_for_update()` + `get_or_create()` with a `created` guard) after verification. Severity is bounded in production by the provider's library-side timestamp tolerance (which limits captured-payload replay to a short window, for example ~5 minutes by default) and by signature verification being an effective access control against forgery. Severity rises when the webhook secret is hardcoded in non-production profiles (making forgery trivial) or when the provider's own legitimate redelivery is counted, since each redelivery re-extends the protected state with no attacker action beyond inducing a retry.
