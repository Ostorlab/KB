The release build enables R8 minification (`isMinifyEnabled = true`), but the ProGuard/R8 configuration ships overly broad `-keep` rules that match whole model packages. A directive such as `-keep class com.example.app.api.models.** { *; }` retains both the class name and every field, method, and constructor from renaming. A second, broader rule such as `-keep class com.example.app.**models.** { *; }` widens the scope to any package whose path contains a `models` segment.

Because R8 deterministically prevents renaming of matched classes and members, the data-class property identifiers declared in those packages survive unobfuscated in the shipped DEX. When the matched models carry sensitive identifiers (for example clinical or billing field names that mirror a live API contract), the release APK leaks a faithful map of the server schema. Anyone decompiling the APK with tools such as JADX or Apktool recovers the original human-readable class and field names, which facilitates API schema inference, payload crafting, and targeting of sensitive data flows.

This is an information disclosure that *facilitates* but does not itself grant unauthorized access. Every API call still requires a valid bearer token and no authentication bypass is introduced by the keep rules. The marginal exposure is the Kotlin model and field identifiers, which overlap with schema information that may also be embedded independently by code generators (for example operation documents baked into generated operation classes).

The root cause is configuration scope, not the absence of obfuscation: R8 is active, but the keep rules disable obfuscation for an entire package tree instead of only for the members that genuinely require their original names at runtime.
