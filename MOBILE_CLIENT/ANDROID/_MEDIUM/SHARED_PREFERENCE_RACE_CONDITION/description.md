A `SharedPreferences.Editor` stages pending mutations into an internal map that is merged into the committed state only on `commit()`/`apply()`. When a single `Editor` instance is created once (for example stored in a long-lived field via `sharedPrefs.edit()`) and then reused by every mutation (`putString`, `putInt`, `putBoolean`, `remove`, `clear`) followed by `commit()`/`apply()`, that pending-write map is shared across all callers without synchronization. A flush performed by one thread can therefore persist values staged by a different thread.

With concurrent writers — for example a background token refresh running on an unstructured `Dispatchers.IO` coroutine, a UI-thread logout writing an empty token, and a migration login all routing through the same editor — this produces a TOCTOU race condition (CWE-362, CWE-364). A successful refresh can be silently discarded leaving the session tokenless, or in the reverse interleaving an explicit logout can be overwritten so the session persists after sign-out. The harmful outcome is timing-dependent and affects session integrity rather than enabling direct token theft.

The per-request read surface is also affected when the token is read directly from shared preferences without caching or locking: concurrent requests fired during an in-flight refresh can read the pre-refresh (possibly expired) value, and with no OkHttp `Authenticator`/401-retry the resulting `RestApiException` is not self-healed.
